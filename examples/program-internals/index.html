part 01 · 作品簡介 & 動機

作品簡介 & 動機

為什麼要讓兩個模型討論出更好的結果——以及手動來回複製貼上的四個問題。

進入 Part 01。先從根本問題開始：為什麼需要 PromptStorm？

平常我們會把 A LLM 的答案貼給 B LLM，再貼回去。這個流程有用，但手動做很痛苦。PromptStorm 就是把這個流程自動化。

作品簡介 & 動機

什麼是 PromptStorm

讓兩個 LLM 在終端機裡討論，使用者當主持人，最終得出比單一模型更好的結果。

核心想法

把「把 A 的答案丟給 B，再把 B 的答案丟回 A」自動化。讓多模型互相激盪，因為不同模型有不同的盲點，討論後答案更完整。

三種互動模式

Discussion — 合作分析，收斂出完整答案（主線）
Debate — 對立觀點，測試論點強度
Dialogue — 角色情境對話

手動做的 4 個問題

  • 流程中斷 — 每輪都要自己整理上下文
  • 容易遺漏 — 討論越長越難維持完整 transcript
  • 方向難控制 — 要手動改 prompt 引導
  • 缺少結論 — 最後還要再請模型總結

PromptStorm 的核心很簡單：讓兩個 LLM 在終端機裡討論，使用者當主持人，最終得出比單一模型更好的結果。

三種模式中 Discussion 是主線——兩方不是互相攻擊，而是從不同角度補足盲點，最後收斂出更好的答案。

手動來回複製貼上有四個問題：流程中斷、容易遺漏、方向難控制、缺少結論。PromptStorm 就是為了解決這些而生的。

Debate — AI 工具：CLI vs GUI 結論

=============================================
          AI 工具：CLI vs GUI 結論
=============================================

▶ 執行摘要
本次辯論雙方各持有理據，未能形成共識，故人類裁決為 TIE。
- CLI 強調可組合、腳本化、在 CI/CD 與自動化流程中的直接可用性。
- GUI 強調多模態輸入、視覺化狀態管理、跨角色（設計、產品、QA）協作的易用性。
兩者皆有不可忽視的優勢，最佳策略應是 **同時提供 CLI 與 GUI，並以統一 API 為核心**。

▶ A 方（CLI 派）核心論點
1. **組合性 & 管道**：`pipe` 讓 AI 輸出直接串接其他工具（grep、sed、git），適合批量處理與腳本自動化。
2. **自動化與規模化**：在 CI/CD、監控腳本中可直接呼叫，無需瀏覽器或手動點擊。
3. **工具鏈一致性**：Unix 哲學——「做一件事，做好」；CLI 天然可被容器、k8s 等環境調用。

▶ B 方（GUI 派）核心論點
1. **多模態與零成本輸入**：支援圖片、JSON、截圖等複合資訊，CLI 需要繁雜編碼與轉義。
2. **視覺化思考迴路**：分支對話、Diff、標註、即時預覽，降低人工驗證 AI 輸出錯誤的成本。
3. **跨角色協作**：產品、設計、QA 等非技術人員可直接在網頁上審核、決策，避免「文字牆」阻斷溝通。

▶ 為何裁定為 TIE 合理
- **需求多樣**：開發者、DevOps 團隊需要腳本化的 CLI；而業務、設計、測試人員則需圖形化的 GUI。
- **技術互補**：CLI 提供高效管道與自動化入口，GUI 提供多模態輸入與可視化決策支援，兩者缺一不可。
- 雙方均未能證明另一方的缺點足以完全抵消其優勢，故平手最為貼切。

▶ 具體後續建議
1. **先行打造統一 HTTP/JSON API**：作為核心服務，CLI 只需 thin wrapper，GUI 只需前端呼叫。
2. **同步開發 CLI 包裝與 Web 前端**  
   - CLI：提供 `ai-tool <command> [options]`，支援 pipe、JSON 輸入/輸出。  
   - GUI：提供多模態上傳、即時 Diff、交互式測試面板，並內建「導出 CLI 命令」功能。
3. **在 CI/CD 中使用 API**：無論是腳本或自動化流程，都透過 API 呼叫 AI，保持環境無瀏覽器依賴。
4. **使用者教育與文件**：針對開發者提供 Bash/Zsh 完整範例；針對非技術角色提供 UI 教學影片。
5. **迭代驗證**：先在小型內部項目中同時測試 CLI 與 GUI 的工作流，收集使用率與效率指標，決定資源分配比例。

---------------------------------------------
結論：同時投資 CLI 與 GUI，圍繞統一 API 建構生態，可兼顧自動化效率與跨職能可用性，最終提升 AI 工具的整體價值。

Debate 模式產出的全文。兩個模型從 CLI 和 GUI 兩方觀點辯論，最後由人類裁定 TIE。

part 02 · 製作過程

Vibe Coding 製作過程

從定義使用情境到補上錯誤處理，五個階段逐步成型。每個階段都在解決一個具體問題。

進入 Part 02。這個專案是典型 vibe coding——先把使用體驗描述清楚，再逐步產生、修改程式碼。

五個階段：定義情境、拆模組、接 API、加控制、補錯誤處理。

vibe coding · 01

先定義使用情境

不是先想架構，而是先確定使用者在終端機裡想做什麼。最基本的流程只有三步：

# 在終端機輸入指令啟動一場討論 $ promptstorm discussion # 輸入角色與主題 Player A > 樂觀創業者 # Player A 的角色人格 Player B > 謹慎投資人 # Player B 的角色人格 Topic > AI 初創該優先營收嗎 # 討論主題 # 系統讓 A/B 輪流發言...
🎯 目標

在終端機快速開一場雙模型討論，得到更好的答案。

⚡ 第一版：先求有

A 說完換 B、B 說完換 A，這樣就好。不需要控制選單、不需要多模式。

🔁 後續疊加

再補控制選單、多模式、錯誤處理、語言偵測、fallback。

第一階段：先定義使用情境。

核心問題很簡單：使用者想在終端機快速開一場兩個模型互相討論的 session，最終得到更好的答案。

最基本的流程只有三步：輸入指令、輸入角色和主題、A 和 B 開始輪流發言。第一版目標就是「先讓它動起來」，後面再逐步疊加。

vibe coding · 02

逐步拆出七個模組

功能變多後，把所有邏輯塞在一個檔案難以維護。每個模組只做一件事：

cli.py

終端機指令、使用者輸入、控制選單、畫面輸出。

只負責 UI
engine.py

回合流程、prompt 組裝、對話紀錄、重試機制。

只負責流程
provider.py

實際呼叫模型 API。

只負責 API
models.py

資料結構定義。

只負責型別
modes.py

三種互動模式的 prompt 與文案。

只負責設定
reporter.py

結論輸出與 fallback。

只負責總結
config.py

API key 與模型設定。支援三層覆蓋。

只負責設定

未來想換模型平台 → 改 provider.py。想加新模式 → 改 modes.py。想換終端機 → 改 cli.py。彼此不干擾。

第二階段：拆模組。七個獨立模組各司其職。

這樣拆的好處是：每個模組的變更不會影響其他模組。想換模型平台只改 provider、想加新模式只改 modes、想換 UI 只改 cli。

這就是單一職責原則在實戰中的應用。

vibe coding · 03

接上 OpenAI SDK + Vercel AI Gateway

OpenAI Python SDK

別人寫好的程式庫，處理 API 請求格式、串流接收、錯誤處理。開發者不用從零刻畫。

Vercel AI Gateway

API 代理層，收到請求後轉發到上游模型（Gemini / Qwen / GPT），結果送回。同一個 SDK 寫法串所有廠商。

# 透過 Vercel AI Gateway 統一轉發到不同廠商的模型 client = OpenAI( # 使用 OpenAI SDK 做為通用客戶端 api_key=key, # Vercel AI Gateway 的 API 金鑰 base_url="https://ai-gateway.vercel.sh/v1" # Gateway endpoint，統一入口 ) # 只需換 model 名稱就能切換上游模型 # model = "google/gemini-...flash-lite" # Gemini：輕量快速的 Player A # model = "alibaba/qwen-3-32b" # Qwen：提供不同觀點的 Player B # model = "openai/gpt-oss-120b" # GPT：品質優先的 Report model
1️⃣
SDK 封裝細節

API 格式、串流、錯誤處理全部封裝在 SDK 裡，開發者只需傳入 model + messages。

2️⃣
Gateway 統一入口

不同廠商模型用同一個 endpoint。靠 model 參數區分上游。

3️⃣
Streaming 即時顯示

stream=True 讓 CLI 逐段印出文字，使用者不用等完整回應。

為什麼不直接連 OpenAI？

因為 Gemini 和 Qwen 的觀點與 GPT 不同——三個模型來自三個廠商，訓練資料與推理方式各異，討論的觀點差異才能最大化。

第三階段：接 API。使用兩層——OpenAI SDK 負責通訊細節，Vercel AI Gateway 負責路由到不同廠商。

三個模型來自三個廠商：Google Gemini、Alibaba Qwen、OpenAI GPT。它們的訓練資料與推理方式都不同，觀點差異最大化，這也是討論能產出更好結果的基礎。

vibe coding · 04

加入控制選單 — 使用者是主持人

一般 chatbot 是一問一答，但 PromptStorm 是「主持一場多模型討論」。使用者可以隨時介入方向：

[A] 我目前支持 A，讓雙方再討論 N 回合 [B] 我目前支持 B，讓雙方再討論 N 回合 [R] 我目前都不支持，讓雙方再討論 N 回合 [I] 我想補充一句話 [O] 輸出結論並結束
A/B/R
支持方向 + 繼續討論

選擇支持方後，系統會問要再討論幾回合。討論會在原 session 上繼續追加，不中斷。

I
人類補充 + 子選單

輸入補充內容後，出現子選單：按 Enter 繼續 1 回合，或輸入 A/B/R/O 改變方向。

O
輸出結論

呼叫 report model（完整 transcript 作為上下文）產出總結，或使用本地 fallback。

第四階段：控制選單。這是 PromptStorm 最重要的 UX 設計。

選單讓使用者當主持人，隨時調整方向。選 A/B/R 會問要繼續幾回合。選 I 後還有子選單——按 Enter 繼續 1 回合，或切換到其他選項。

這是 PromptStorm 和「兩個 bot 自動聊天」最大的差異。

vibe coding · 05

補上錯誤處理與安全機制

實際呼叫 LLM API 時可能遇到各種問題。程式加入三層保護：

🔄 Rate Limit 重試

遇到 429（rate limit）時自動重試。其他錯誤直接拋出——設定錯了不浪費時間傻等。

# 只有 rate limit (429) 才重試，其他錯誤直接拋出 if attempts >= retries or not is_rate_limit(error): raise # 超過重試次數 or 非 rate limit → 放棄

💾 模型失敗不遺失資料

某一輪模型呼叫失敗時，已產生的所有 transcript 全部保存。使用者的討論不會因為一次錯誤就消失。

📄 Conclusion Fallback

最後總結的 report model 失敗時，不崩潰。改用本地 fallback：說明錯誤 + 顯示 final state + 顯示完整 transcript。

另外，429 錯誤訊息會自動摘要縮短，不會洗滿 terminal。錯誤類型區分：rate limit（可重試）vs 其他（直接中斷）vs conclusion（走 fallback）。

第五階段：錯誤處理。三層保護從 API 到結論全涵蓋。

Rate limit 重試：只有 429 才重試，避免設定錯誤時無意義等待。模型失敗不遺失資料：已產生的 transcript 都留著。Conclusion fallback：最關鍵的保障——使用者討論了半小時，不會因為最後一步出錯而白費。

pipeline · 完整執行流程

從指令到結論，13 步一氣呵成

1 使用者輸入 promptstorm discussion 2 cli.main() 用 argparse 解析指令 3 run_session() 載入設定與 API key 4 輸入 Player A persona、Player B persona、Topic 5 建立 VercelGatewayProvider 和 DebateEngine 6 engine.run() 建立 DebateSession 7 _run_rounds() 讓 A/B 輪流發言（雙層迴圈） 8 每次發言用 _build_messages() 組 prompt（含完整歷史） 9 provider.complete() 串流呼叫模型 10 clean_response() 清理模型輸出雜訊 11 新增 DebateTurn 到 session.turns 12 run_control_loop() 讓使用者決定下一步 13 ConclusionWriter 輸出 conclusion 或 synthesis

13 步一氣呵成。run_session() 是導演，串起 Config、Engine、Provider、Reporter。

第 7 步是核心——_run_rounds() 的雙層迴圈。第 8 步的 _build_messages() 把完整歷史塞進 prompt，這也是兩個 LLM 能真正對話的關鍵。

不可變物件確保設定不會在執行過程中被意外修改。模式定義是唯讀的。

15 個欄位一次打包

VercelGatewayProvider — 實際實作

# 實際串接 Vercel AI Gateway 的 Provider 實作 class VercelGatewayProvider: def __init__(self, api_key): self._client = None # 延遲初始化（Lazy Init），不在一開始建立連線 def _get_client(self): # 取得或建立 OpenAI 客戶端 if self._client is None: # 首次呼叫才初始化 self._client = OpenAI( api_key=self.api_key, base_url="https://ai-gateway.vercel.sh/v1") # Gateway 統一入口 return self._client def complete(self, model, messages, ...): # 實作 Protocol 的 complete() client = _get_client() # 取得延遲初始化的 client stream = client.chat.completions.create( model=model, messages=messages, stream=True) # 串流模式接收 # 逐段收集輸出，透過 callback 即時回傳給 CLI 顯示
⚡
延遲初始化（Lazy Init）

client 不是 __init__ 就建立，等第一次呼叫 complete() 才初始化。避免開局浪費資源。

📡
Streaming 實作細節

SDK 的 stream=True → 串流模式接收。每收到一段 chunk 就透過 callback 送回 CLI 顯示。使用者看到的是逐字出現的文字，不是一次跳一整段。

🔌
同一個 SDK 不同 model

client 是同一個 OpenAI 實例，只換 model 參數就切換上游。Gemini / Qwen / GPT 共用同一條連線。

實際實作 VercelGatewayProvider 的細節。

延遲初始化：client 在第一次 complete() 才建立，不是 __init__ 時。Streaming：stream=True 讓 SDK 以串流接收，CLI 逐段印出。

同一個 OpenAI 實例、同一個 base_url，只換 model 參數就切不同廠商模型。

Prompt 組裝 + Transcript 格式

# _build_messages() — 組裝發送給模型的 prompt if transcript is None: # 第一輪：還沒有歷史紀錄 # 第一輪：使用 opening_instruction 開場 system = profile.system_instruction # 系統提示（角色設定） user = profile.opening_instruction + topic # 開場引導 + 主題 else: # 第 N 輪：已有討論歷史 # 第 N 輪：使用 continuation 指令 + 包含完整討論歷史 system = profile.system_instruction # 系統提示不變 user = profile.continuation_instruction # 接續引導（如「反駁對方論點」） user += transcript # 完整歷史——A 和 B 能真正對話的關鍵
# _format_transcript() — 把結構化的 DebateTurn 列表轉成純文字給模型閱讀 Round 1 [A: Point of View A] ... # 格式：回合 [發言者: 內容] Round 1 [B: Point of View B] ... # 每輪 A 和 B 各一次 Human input after Round 1: ... # 使用者的補充意見也一併記錄
1️⃣
第一輪 vs 第 N 輪

第一輪用 opening_instruction 開場引導。後續用 continuation_instruction 要求接續，兩種指令來自 ModeProfile。

2️⃣
transcript 串接歷史

完整討論歷史附加在 user 訊息末端。模型看到的不只是當下的 prompt，而是整場討論演進。

3️⃣
結構化 → 純文字

_format_transcript() 把 DebateTurn list 轉成人類/模型都好讀的純文字格式，回合、發言者、內容一目瞭然。

_build_messages() 組裝 prompt。關鍵差異在第一輪 vs 第 N 輪：第一輪用 opening_instruction，後續用 continuation_instruction。

_format_transcript() 是把結構化的發言記錄轉成純文字，放入 prompt 讓模型能讀懂前面發生什麼。

run_session() — 整場 session 的導演

# run_session() — 整場 session 的導演，11 步串起所有模組 def run_session(mode_name): 1 profile = get_mode_profile(mode_name) # 取得模式設定檔 2 config = load_config() # 載入設定 3 if not config.api_key: get_api_key() # 沒有 API key 就引導設定 4 persona_a = input("Player A: ") # 輸入 A 角色 5 persona_b = input("Player B: ") # 輸入 B 角色 6 topic = input("Topic: ") # 輸入討論主題 7 provider = VercelGatewayProvider(config.api_key) # 建立 Provider 8 engine = DebateEngine(provider, ...) # 建立 Engine（傳入 callback） 9 engine.run(persona_a, persona_b, topic) # 開始討論 10 run_control_loop(engine) # 控制迴圈（使用者決定下一步） 11 writer.generate_conclusion() # 輸出結論
🎬
run_session() 做的事

取得 ModeProfile → 載入設定 → 取得 API key → 輸入角色主題 → 建立 Provider → 建立 Engine → 設定 callback → engine.run() → 控制迴圈 → 結論輸出。整場 session 全由它串聯。

Callback 設定

CLI 在建立 engine 時傳入 callback：on_turn_start 印出回合標題，on_response 即時顯示模型文字，on_turn_end 印出分隔線與錯誤提示。

run_session() 是整場 session 的導演，11 步把 Config、Engine、Provider、Reporter 全部串起來。可以說是 CLI 裡最重要的一隻函式。

Callback 設定讓 engine 和 CLI 分離——engine 只管邏輯，CLI 只管顯示。

Fallback 的設計哲學

使用者花半小時討論的內容，不會因為最後一步 API 出錯而白費。不管 report model 怎麼掛，已產生的 debate 內容一定保得住。

心得 · 01

從交易機器人到 PromptStorm

一開始，我們團隊野心勃勃，想打造一款能帶來「最佳年化報酬率」的自動交易機器人。但很快我們就嚐到了苦頭。因為缺乏真實的金融市場經驗，「追求高報酬」對我們而言變成了一個極度空洞的目標。無論我們怎麼用歷史數據回測、反覆調整參數，市場裡不可控的因素實在太多。專案陷入了死胡同。

這場挫敗讓我們停下來反思：與其追逐一個不熟悉的虛幻數字，不如回頭去解決生活中真正遇到的痛點。我們觀察到很多人會手動把一個 AI 的結論貼給另一個 AI 反駁——既然大家都在手動做這件事，為什麼不乾脆打造一個系統自動化？

這次的跌跌撞撞讓我學到最深刻的一課：最值錢的從來不是工具本身，而是你的「核心想法」。如果目標空洞，AI 只會放大你的迷茫；但當你精準抓到一個真實的痛點時，AI 就能成為解決問題的最強助力。

第一段心得：從失敗的交易機器人專案中轉向，找到真實痛點。核心想法比工具更重要。

心得 · 02

流程設計比 Prompt 更關鍵

做 PromptStorm 的過程讓我發現，這個專案不只是「讓兩個 AI 聊天」，而是比較像在做一個小型的對話系統。

一開始以為重點是 prompt，但實際做下去才知道，真正重要的是「流程設計」：要怎麼讓 A、B 兩個模型輪流發言、怎麼把過去對話保存起來、怎麼讓使用者可以中途插手改方向，這些比單純寫 prompt 更關鍵。

另外也學到模型其實很容易跑掉，所以需要額外做整理和控制，例如清理輸出、統一格式、把對話記錄整理成 transcript，才能讓整體討論變穩定。

最有感的是 CLI 不只是輸入輸出工具，而是像一個「主持人介面」，使用者可以決定要支持哪一方、要不要繼續、或直接要求結論，整個互動變得更有控制感。

總結來說，AI 應用的重點不是模型本身，而是怎麼設計一個讓模型可以好好合作的系統。

第二段心得：流程設計比 prompt 更關鍵。系統設計 > 模型本身。
