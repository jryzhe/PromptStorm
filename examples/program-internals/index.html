01

part 01 · 作品簡介 & 動機

作品簡介 & 動機

為什麼要讓兩個模型討論出更好的結果——以及手動來回複製貼上的四個問題。

進入 Part 01。先從根本問題開始：為什麼需要 PromptStorm？

平常我們會把 A LLM 的答案貼給 B LLM，再貼回去。這個流程有用，但手動做很痛苦。PromptStorm 就是把這個流程自動化。

作品簡介 & 動機

什麼是 PromptStorm

讓兩個 LLM 在終端機裡討論，使用者當主持人，最終得出比單一模型更好的結果。

核心想法

把「把 A 的答案丟給 B，再把 B 的答案丟回 A」自動化。讓多模型互相激盪，因為不同模型有不同的盲點，討論後答案更完整。

三種互動模式

Discussion — 合作分析，收斂出完整答案（主線）
Debate — 對立觀點，測試論點強度
Dialogue — 角色情境對話

手動做的 4 個問題

  • 流程中斷 — 每輪都要自己整理上下文
  • 容易遺漏 — 討論越長越難維持完整 transcript
  • 方向難控制 — 要手動改 prompt 引導
  • 缺少結論 — 最後還要再請模型總結

PromptStorm 的核心很簡單：讓兩個 LLM 在終端機裡討論，使用者當主持人，最終得出比單一模型更好的結果。

三種模式中 Discussion 是主線——兩方不是互相攻擊，而是從不同角度補足盲點，最後收斂出更好的答案。

手動來回複製貼上有四個問題：流程中斷、容易遺漏、方向難控制、缺少結論。PromptStorm 就是為了解決這些而生的。

part 02 · Vibe Coding 製作過程

Vibe Coding 製作過程

從定義使用情境到補上錯誤處理，五個階段逐步成型。每個階段都在解決一個具體問題。

進入 Part 02。這個專案是典型 vibe coding——先把使用體驗描述清楚，再逐步產生、修改程式碼。

五個階段：定義情境、拆模組、接 API、加控制、補錯誤處理。

vibe coding · 01

先定義使用情境

不是先想架構，而是先確定使用者在終端機裡想做什麼。最基本的流程只有三步：

# 在終端機輸入指令啟動一場討論 $ promptstorm discussion # 輸入角色與主題 Player A > 樂觀創業者 # Player A 的角色人格 Player B > 謹慎投資人 # Player B 的角色人格 Topic > AI 初創該優先營收嗎 # 討論主題 # 系統讓 A/B 輪流發言...
🎯 目標

在終端機快速開一場雙模型討論，得到更好的答案。

⚡ 第一版：先求有

A 說完換 B、B 說完換 A，這樣就好。不需要控制選單、不需要多模式。

🔁 後續疊加

再補控制選單、多模式、錯誤處理、語言偵測、fallback。

第一階段：先定義使用情境。

核心問題很簡單：使用者想在終端機快速開一場兩個模型互相討論的 session，最終得到更好的答案。

最基本的流程只有三步：輸入指令、輸入角色和主題、A 和 B 開始輪流發言。第一版目標就是「先讓它動起來」，後面再逐步疊加。

vibe coding · 02

逐步拆出七個模組

功能變多後，把所有邏輯塞在一個檔案難以維護。每個模組只做一件事：

cli.py

終端機指令、使用者輸入、控制選單、畫面輸出。

只負責 UI
engine.py

回合流程、prompt 組裝、對話紀錄、重試機制。

只負責流程
provider.py

實際呼叫模型 API。

只負責 API
models.py

資料結構定義。

只負責型別
modes.py

三種互動模式的 prompt 與文案。

只負責設定
reporter.py

結論輸出與 fallback。

只負責總結
config.py

API key 與模型設定。支援三層覆蓋。

只負責設定

未來想換模型平台 → 改 provider.py。想加新模式 → 改 modes.py。想換終端機 → 改 cli.py。彼此不干擾。

第二階段：拆模組。七個獨立模組各司其職。

這樣拆的好處是：每個模組的變更不會影響其他模組。想換模型平台只改 provider、想加新模式只改 modes、想換 UI 只改 cli。

這就是單一職責原則在實戰中的應用。

vibe coding · 03

接上 OpenAI SDK + Vercel AI Gateway

OpenAI Python SDK

別人寫好的程式庫，處理 API 請求格式、串流接收、錯誤處理。開發者不用從零刻畫。

Vercel AI Gateway

API 代理層，收到請求後轉發到上游模型（Gemini / Qwen / GPT），結果送回。同一個 SDK 寫法串所有廠商。

# 透過 Vercel AI Gateway 統一轉發到不同廠商的模型 client = OpenAI( # 使用 OpenAI SDK 做為通用客戶端 api_key=key, # Vercel AI Gateway 的 API 金鑰 base_url="https://ai-gateway.vercel.sh/v1" # Gateway endpoint，統一入口 ) # 只需換 model 名稱就能切換上游模型 # model = "google/gemini-...flash-lite" # Gemini：輕量快速的 Player A # model = "alibaba/qwen-3-32b" # Qwen：提供不同觀點的 Player B # model = "openai/gpt-oss-120b" # GPT：品質優先的 Report model
1️⃣
SDK 封裝細節

API 格式、串流、錯誤處理全部封裝在 SDK 裡，開發者只需傳入 model + messages。

2️⃣
Gateway 統一入口

不同廠商模型用同一個 endpoint。靠 model 參數區分上游。

3️⃣
Streaming 即時顯示

stream=True 讓 CLI 逐段印出文字，使用者不用等完整回應。

為什麼不直接連 OpenAI？

因為 Gemini 和 Qwen 的觀點與 GPT 不同——三個模型來自三個廠商，訓練資料與推理方式各異，討論的觀點差異才能最大化。

第三階段：接 API。使用兩層——OpenAI SDK 負責通訊細節，Vercel AI Gateway 負責路由到不同廠商。

三個模型來自三個廠商：Google Gemini、Alibaba Qwen、OpenAI GPT。它們的訓練資料與推理方式都不同，觀點差異最大化，這也是討論能產出更好結果的基礎。

vibe coding · 04

加入控制選單 — 使用者是主持人

一般 chatbot 是一問一答，但 PromptStorm 是「主持一場多模型討論」。使用者可以隨時介入方向：

[A] 我目前支持 A，讓雙方再討論 N 回合 [B] 我目前支持 B，讓雙方再討論 N 回合 [R] 我目前都不支持，讓雙方再討論 N 回合 [I] 我想補充一句話 [O] 輸出結論並結束
A/B/R
支持方向 + 繼續討論

選擇支持方後，系統會問要再討論幾回合。討論會在原 session 上繼續追加，不中斷。

I
人類補充 + 子選單

輸入補充內容後，出現子選單：按 Enter 繼續 1 回合，或輸入 A/B/R/O 改變方向。

O
輸出結論

呼叫 report model（完整 transcript 作為上下文）產出總結，或使用本地 fallback。

第四階段：控制選單。這是 PromptStorm 最重要的 UX 設計。

選單讓使用者當主持人，隨時調整方向。選 A/B/R 會問要繼續幾回合。選 I 後還有子選單——按 Enter 繼續 1 回合，或切換到其他選項。

這是 PromptStorm 和「兩個 bot 自動聊天」最大的差異。

vibe coding · 05

補上錯誤處理與安全機制

實際呼叫 LLM API 時可能遇到各種問題。程式加入三層保護：

🔄 Rate Limit 重試

遇到 429（rate limit）時自動重試。其他錯誤直接拋出——設定錯了不浪費時間傻等。

# 只有 rate limit (429) 才重試，其他錯誤直接拋出 if attempts >= retries or not is_rate_limit(error): raise # 超過重試次數 or 非 rate limit → 放棄

💾 模型失敗不遺失資料

某一輪模型呼叫失敗時，已產生的所有 transcript 全部保存。使用者的討論不會因為一次錯誤就消失。

📄 Conclusion Fallback

最後總結的 report model 失敗時，不崩潰。改用本地 fallback：說明錯誤 + 顯示 final state + 顯示完整 transcript。

另外，429 錯誤訊息會自動摘要縮短，不會洗滿 terminal。錯誤類型區分：rate limit（可重試）vs 其他（直接中斷）vs conclusion（走 fallback）。

第五階段：錯誤處理。三層保護從 API 到結論全涵蓋。

Rate limit 重試：只有 429 才重試，避免設定錯誤時無意義等待。模型失敗不遺失資料：已產生的 transcript 都留著。Conclusion fallback：最關鍵的保障——使用者討論了半小時，不會因為最後一步出錯而白費。

module 01 · models.py

四組核心資料結構

# 定義設定檔的資料結構 @dataclass class PromptStormConfig: api_key: str # Vercel AI Gateway 的 API 金鑰 player_a_model: str # Player A 使用的模型名稱 player_b_model: str # Player B 使用的模型名稱 report_model: str # 負責產出最終報告/結論的模型 # 統一包裝模型回傳文字的結構 @dataclass class ModelResponse: text: str # 模型生成的純文字內容 # 記錄單一回合發言的詳細資料 @dataclass class DebateTurn: session_id: str # 該場次唯一的 Session ID round: int # 第幾回合（從 1 開始） speaker: str # 發言者身分：A / B / USER persona: str # 發言者設定的角色人格名稱 model: str # 生成該次回應的模型 response_text: str # 發言的實際文字內容 timestamp: str # 發言時間戳記 status: str = "ok" # 呼叫狀態：ok / error error: str | None = None # 失敗時儲存錯誤訊息 # 整場討論的容器——一份完整會議紀錄 @dataclass class DebateSession: session_id: str # 這場討論的唯一 ID timestamp: str # 討論開始的時間 player_a: str # Player A 的角色人格 player_b: str # Player B 的角色人格 topic: str # 討論主題 turns: list[DebateTurn] # 所有回合的發言紀錄
PromptStormConfig

三種模型可分開設定。討論模型和總結模型可以不同——輕量模型討論、強大模型總結。

ModelResponse

抽象層關鍵。engine 不知道底層 API 格式，只處理 ModelResponse。

DebateTurn

一次發言的最小單位。不管是模型還是使用者，格式都一樣。

DebateSession

整場討論的容器，一份完整會議紀錄。

第一個模組：models.py。四個 dataclass 定義了所有資料結構。

PromptStormConfig 的三個 model 欄位可以分開設，這是策略性的——用輕量模型討論、用強大模型總結。

ModelResponse 很簡單但很重要——它是 adapter 模式的核心。DebateTurn 是 transcript 的基本單位。DebateSession 是容器。

module 01 · models.py

工具函式 + Session ID 格式

normalize_verdict()

把使用者最後的選擇標準化：A 和 B 保持不變，C / TIE / DRAW 統一視為 TIE。後面的 conclusion writer 不用處理多種輸入格式。

# 標準化使用者的最終選擇，統一格式供 conclusion writer 使用 def normalize_verdict(raw_value: str) -> str: value = raw_value.strip().upper() # 去除空白並轉大寫 if value in {"A", "B"}: # A 或 B 保持原樣 return value if value in {"C", "TIE", "DRAW"}: # C / TIE / DRAW 統一為 TIE return "TIE" raise ValueError("Verdict must be A, B, or C.") # 非預期輸入直接報錯
Session ID 格式

YYYYMMDD-HHMMSS-{8位UUID hex}
例如：20260613-153022-a1b2c3d4

📌
為什麼要這個格式？

時間戳記在前，方便排序與辨識 session。UUID 避免同秒多 session 衝突。除錯時一眼看出 session 是何時建立的。

📌
status + error 欄位的作用

每個 turn 有自己的 status（ok / error）和 error message。後續除錯時可以精確定位哪一輪、哪個模型出了問題。

工具函式 normalize_verdict——標準化使用者選擇。把 C、TIE、DRAW 統一為 TIE。

Session ID 格式 YYYYMMDD-HHMMSS-8位UUID hex。時間在前面方便排序，UUID 避免衝突。

DebateTurn 的 status 和 error 欄位不是裝飾——除錯時可以精確定位哪一輪、哪個模型出了問題。

module 03 · modes.py

ModeProfile — 模式的抽象層

# 模式設定檔——三種模式的所有行為差異封裝在此 @dataclass(frozen=True) # frozen=True 確保設定不可修改 class ModeProfile: name: str # 模式名稱：discussion / debate / dialogue help_text: str # 使用說明文字 title: str # 顯示標題 identity_label: str # 自己的身分稱呼 counterpart_label: str # 對方的稱呼 default_persona_template: str # 預設角色模板 system_instruction: str # 模型的系統提示 opening_instruction: str # 第一輪的開場指令 continuation_instruction: str # 後續回合的接續指令 support_contexts: Mapping[str, str] # 支援/不支援的立場文案 control_lines: tuple[str, ...] # 控制選單的文字行 output_label: str # 結論的標籤 final_state_label: str # 最終狀態的標籤 conclusion_system: str # 結論階段的系統提示 conclusion_instruction: str # 結論階段的指令
為什麼是 frozen dataclass？

不可變物件確保設定不會在執行過程中被意外修改。模式定義是唯讀的。

engine 不需要知道自己跑哪一種模式

ModeProfile 把所有差異封裝在設定裡。engine 只要拿到 profile，開場用 opening_instruction、接續用 continuation_instruction，不 care 是 debate 還是 dialogue。

📦
15 個欄位一次打包

從 UI 文案到 prompt 指令到 conclusion 設定，全部在一個結構裡。要加新模式就加一組設定。

engine 不需要知道自己正在跑 debate 還是 discussion。它只要拿到 profile，用 opening_instruction 或 continuation_instruction 就好。這是策略模式的實戰應用。

語言偵測：自動判斷輸出語言

# 自動判斷輸出語言——確保模型用使用者輸入的語言回答 def detect_output_language( topic, human_input ) -> str: # 規則 1：文字明確要求英文 → English # 規則 2：文字明確要求中文 → Traditional Chinese # 規則 3：中文字元比例高 → Traditional Chinese # 規則 4：英文字母較多 → English
為什麼需要語言偵測？

模型預設傾向用英文回應。但如果使用者用中文輸入主題，討論理應輸出中文。語言偵測確保模型回應貼近使用者輸入語言。

🌐
實作方式

先檢查文字是否明確指定語言（「用中文回答」），再按字符比例判斷。輕量、無外部依賴。

語言偵測函式 detect_output_language() 解決一個實際問題：模型預設傾向用英文回應。

如果使用者用中文輸入主題，討論理應輸出中文。偵測方式分兩層：先看是否有明確語言要求，再看字符比例。

ModelProvider Protocol — 抽象介面

# ModelProvider Protocol——只要實作 complete() 就能當成 provider 使用 class ModelProvider(Protocol): # Python 結構子型別，不需繼承基底類別 def complete( self, model: str, # 要呼叫的模型名稱 messages: Sequence[dict], # 對話歷史 + 系統提示 ) -> ModelResponse: ... # 回傳統一格式的 ModelResponse
結構子型別（Structural Subtyping）

不需要繼承基底類別，不需要註冊。任何類別只要有 complete() 方法且簽章吻合，engine 就能用。

✅
測試性

測試時 inject mock provider，直接回傳固定 ModelResponse。不需真實 API key，不需網路。

✅
可替換性

明天想換 Anthropic？寫一個 AnthropicProvider 實作 complete() 就好。engine、cli 完全不用改。

provider.py 定義了 ModelProvider Protocol。這是 Python 的結構子型別——只要有 complete() 方法就能用。

最大的好處是測試性。寫測試時 inject 一個 mock provider，不用真的打 API。

VercelGatewayProvider — 實際實作

# 實際串接 Vercel AI Gateway 的 Provider 實作 class VercelGatewayProvider: def __init__(self, api_key): self._client = None # 延遲初始化（Lazy Init），不在一開始建立連線 def _get_client(self): # 取得或建立 OpenAI 客戶端 if self._client is None: # 首次呼叫才初始化 self._client = OpenAI( api_key=self.api_key, base_url="https://ai-gateway.vercel.sh/v1") # Gateway 統一入口 return self._client def complete(self, model, messages, ...): # 實作 Protocol 的 complete() client = _get_client() # 取得延遲初始化的 client stream = client.chat.completions.create( model=model, messages=messages, stream=True) # 串流模式接收 # 逐段收集輸出，透過 callback 即時回傳給 CLI 顯示
⚡
延遲初始化（Lazy Init）

client 不是 __init__ 就建立，等第一次呼叫 complete() 才初始化。避免開局浪費資源。

📡
Streaming 實作細節

SDK 的 stream=True → 串流模式接收。每收到一段 chunk 就透過 callback 送回 CLI 顯示。使用者看到的是逐字出現的文字，不是一次跳一整段。

🔌
同一個 SDK 不同 model

client 是同一個 OpenAI 實例，只換 model 參數就切換上游。Gemini / Qwen / GPT 共用同一條連線。

實際實作 VercelGatewayProvider 的細節。

延遲初始化：client 在第一次 complete() 才建立，不是 __init__ 時。Streaming：stream=True 讓 SDK 以串流接收，CLI 逐段印出。

同一個 OpenAI 實例、同一個 base_url，只換 model 參數就切不同廠商模型。

# _build_messages() — 組裝發送給模型的 prompt if transcript is None: # 第一輪：還沒有歷史紀錄 # 第一輪：使用 opening_instruction 開場 system = profile.system_instruction # 系統提示（角色設定） user = profile.opening_instruction + topic # 開場引導 + 主題 else: # 第 N 輪：已有討論歷史 # 第 N 輪：使用 continuation 指令 + 包含完整討論歷史 system = profile.system_instruction # 系統提示不變 user = profile.continuation_instruction # 接續引導（如「反駁對方論點」） user += transcript # 完整歷史——A 和 B 能真正對話的關鍵
1️⃣
_run_rounds() 雙層迴圈

外層 = 回合數，內層 = A→B 依序。每個 speaker 流程完全統一，不會出現 A 和 B 處理不一致。

2️⃣
transcript 是共享上下文

這就是兩個 LLM 能真正對話的關鍵——每輪模型看到的不只是 topic，而是包含前面所有人的完整歷史。B 知道 A 說了什麼，A 知道 B 回了什麼。

3️⃣
opening vs continuation

第一輪用 opening_instruction 開場。後續用 continuation_instruction 要求接續。兩種 instruction 都來自 ModeProfile。

# _format_transcript() — 把結構化的 DebateTurn 列表轉成純文字給模型閱讀 Round 1 [A: Point of View A] ... # 格式：回合 [發言者: 內容] Round 1 [B: Point of View B] ... # 每輪 A 和 B 各一次 Human input after Round 1: ... # 使用者的補充意見也一併記錄

_build_messages() 的 transcript 機制是 PromptStorm 的靈魂——每輪模型看到完整歷史。B 知道 A 說了什麼，A 知道 B 回了什麼。這不是兩個獨立 chatbot，而是一場真正的對話。

module 05 · engine.py

雜訊清理 + Rate Limit 重試

# clean_response() — 移除模型輸出中的 4 類雜訊 # 類型 1：移除 ... 推理區塊（模型內心 OS，不該讓使用者看到） # 類型 2：移除 "Round 1 [A: ...]" 這類多餘前綴 # 類型 3：移除「好的，我明白你的意思」這類浪費版面的禮貌開場 # 類型 4：Dialogue 模式——移除角色名、括號動作、敘述，只保留台詞

不同模型常輸出不需要的內容。clean_response() 統一清理，讓 terminal 顯示的是乾淨的發言內容。

# _complete_with_retries() — 聰明重試，不浪費時間 if attempts >= self.rate_limit_retries: # 超過重試次數上限 raise # 放棄，往上拋 if not _is_rate_limit_error(error): # 不是 rate limit 錯誤 raise # 非 rate limit 直接拋出（API key 錯了重試也沒用） # 只有 HTTP 429 Too Many Requests 才重試
🧹
clean_response 的價值

<think> 區塊是推理過程，不該讓觀眾看到。「好的，我明白你的意思」這類開場白浪費版面。clean_response 讓發言內容更精煉。

🔄
重試的判斷邏輯

只有 HTTP 429 Too Many Requests 才重試。如果是 401（API key 錯誤）或 400（請求格式錯誤），直接拋出——設定錯了重試也沒用。

⚠️
非重試錯誤的處理

直接 raise 到上一層的 _run_rounds()，然後往上傳到 run()。run() 的 caller（cli.py）負責 catch 並顯示錯誤摘要給使用者。

clean_response 處理四類雜訊：think 推理區塊、多餘前綴、禮貌開場、dialogue 多餘敘述。

_complete_with_retries 的判斷邏輯很重要——只有 rate limit 才重試。API key 錯了重試一百次也沒用，直接拋出讓使用者修正。
